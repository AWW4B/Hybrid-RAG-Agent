Assignment 4 — Work Division

RAG pipeline · Tool orchestrator · Performance & testing

AW

Rayan — RAG pipeline + system integration

Owns: main.py · routes.py · config.py · context.py · Docker

Offline document indexer new

Write indexer.py that walks 50–100 .txt docs, chunks them at 512 tokens with 50-token overlap, embeds with MiniLM-L6-v2, and stores in ChromaDB. Must be re-runnable and report chunk count per doc.

Retriever class + context injection new

Build retriever.py with embedding cache (LRU) so repeated queries skip re-embedding. Wire it into the conversation manager so top-4 chunks are injected before LLM generation. Implement truncation strategy so RAG context never overflows the context window.

Docker + environment updates extend

Update docker-compose.yml to mount ./chroma_db, ./docs, and crm_data.json. Add ChromaDB and sentence-transformers to requirements.txt. Verify the full stack builds cleanly with new dependencies.

Document corpus (50+ docs) new

Curate or write 50–100 Daraz-domain .txt files: product FAQs, return policy, shipping zones, payment methods, seller guide, promo rules. These must be coherent enough that retrieval actually helps the LLM answer questions differently than without RAG.

Estimated effort: heavy — indexer correctness and context truncation logic are the hardest parts to get right

UW

Uwaid — performance, testing, and README

Owns: engine.py · STT/LLM/TTS pipeline · models/

End-to-end latency benchmarking extend

Extend benchmark_full.py to separately measure: retrieval time, tool call latency, and total end-to-end time for RAG queries vs plain queries vs tool-invoking queries. Report p50/p95/min/max for each. Identify and document the bottleneck.

Embedding cache + response caching new

Implement LRU cache (Python functools.lru_cache or Redis) for query embeddings and for tool results that don't change rapidly (e.g. weather cached for 10 min). Measure before/after latency impact and report the numbers.

Concurrent user stress test extend

Update locustfile.py to simulate at least 2 concurrent users each triggering RAG + a tool call simultaneously. Verify session isolation still holds (CRM data for user A must not bleed into user B). Report throughput and any degradation.

Postman collection update + video demo extend

Add Postman requests that exercise: a RAG-grounded answer (verify the response references a doc), a CRM tool call (get + update), and two other tools. Record the 3–5 min demo video showing all three scenarios live.

README for A4 new

Write all required README sections: architecture diagram, document collection details, tools description with input schemas and example invocations, real-time optimisation explanation with actual benchmark numbers, and known limitations. This requires pulling real data from Rayan and Awwab's implementations — cannot be written until the system works.

Estimated effort: significant — benchmarking meaningfully and writing a complete README based on real numbers takes more time than it looks

RY

Awwab — tool orchestrator + all 4 tools

Owns: frontend/ · database.py · locustfile.py · benchmark_full.py

Tool orchestrator with function-calling loop new

Build orchestrator.py with tool registration, JSON schema exposure, <tool_call> tag parsing, async dispatch with 5 s timeout, and error handling. Wire the two-pass loop into engine.py: first LLM pass may emit a tool call; result is fed back for the final streamed response.

CRM tool (mandatory) new

Implement crm.py with get_user_info / update_user_info / list_users backed by a JSON file with SQLite fallback. The LLM must be able to call this to greet returning users by name and recall past preferences.

Weather tool new

Implement weather.py using OpenWeatherMap free tier via httpx. Handle network errors, invalid city names, and timeouts gracefully — return a plain-language string the LLM can incorporate naturally.

Calculator + product search tools new

Calculator: safe eval with whitelist (no builtins). Product search: query the local SQLite product catalogue (already used for sessions) with fuzzy matching. Both must be registered with full JSON schemas so the LLM knows when to call them.

System prompt engineering for tool + RAG use extend

Redesign the system prompt in engine.py to (a) expose tool schemas to the model, (b) instruct it to emit <tool_call> JSON when appropriate, and (c) ground answers in retrieved chunks. This requires careful prompt tuning — Qwen 2.5 needs explicit formatting instructions to reliably produce parseable tool calls.

Estimated effort: heaviest — the two-pass LLM loop and reliable tool-call parsing are the highest-risk parts of the whole assignment
